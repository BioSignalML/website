Ad Hoc Formats

Some software packages claim to use a standard format for signal data when in fact the package developers have implemented their interpretation of a specification and not tested the implementation for standard's compliance.

This hampers the easy interchange of files between and within research groups...

Minerva

Minerva departs from the EDF specification in two ways:
  1. It uses Nul characters (x00) as a pad/fill character in the file header instead of a space character (x20).
  2. The physical minimum/maximum fields are right justified when in fact they should be left justified (Minerva correctly left justifies all other numeric fields).
The EDF specification clearly states that only Ascii printing characters are allowed in a file header and that fields are to be left justified.

Both are easy enough to fix (in fact case 2 just needs a format changed from "%8.1f" to "%-8.1f" although no one has considered fields overflowing with large numbers..). A better approach would be for Minerva to use a standard compliant library -- in both cases historical signal files will need converting...

The EDF/EDF+ format is also in the process of becoming a European Standard. Are there implications for F&P of it using software purporting to use a standard file format when in fact the software doesn't??

For BioSignalML I will use the standard EDF/EDF+ library and have a separate driver to read Minerva's data files (and no longer refer to them as being EDF).
